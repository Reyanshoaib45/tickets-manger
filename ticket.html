P1

TICKET: ARCH-001

Logical Separation of Task Board and Repair Board
Architecture User Experience
Description

The interlinking of the general Task Board and the specialized Repair Board is creating significant user confusion. Users see unrelated items. The philosophy is to isolate functionality until a unified, professional UI/UX can be designed.

Requirements
  1. Repair Dashboard Isolation: Only users with "Technician" role can view "Repair Jobs" page
  2. Task Board Universality: All users have access to general Task Board
  3. Future-Proofing: System must allow future feature flag to merge/hide boards per role

TICKET: WF-002

Part Ordering Workflow - Creation & Assignment
Core Workflow
Description

When a technician determines a part is needed, they must create a "Part Order Request" as a task for another user (e.g., store person). This is a critical path for repair completion.

Requirements
  1. Initiation Point: From repair "In-Progress", against specific Job
  2. Order Modal: Popup with Part Link, Estimated Price, Assign To User, Notes
  3. Task Generation: Creates Task on general Task Board with auto-generated title
  4. Repair Status: Parent repair moves to visual state "Awaiting Parts"

TICKET: PERM-003

Task Board Permissions & Reassignment Logic
Permissions Security
Description

Permissions on the Task Board are currently ambiguous. The decision is to mimic standard project management tools (Jira, ClickUp): users can create tasks for others and manage their own, but cannot arbitrarily move others' tasks.

Requirements
  1. Create & Delete: User can create tasks for any user and delete tasks they created
  2. Move/Restrict: User cannot move a task if not the assignee or creator
  3. Self-Assignment: Any user can assign a task to themselves from task detail view
  4. Audit Log: All assignment changes logged with timestamp and user ID

TICKET: NOTIF-004

Comprehensive Notification System
Feature
Description

Users are operating in a shared environment and need to be informed of actions relevant to them. Notifications should be timely and reduce the need for constant manual checking.

Requirements
  1. Email Notifications for Tasks: Task Assigned, Task Status Changed, Task Deleted
  2. Email Notifications for Part Orders: Part Order Received sent to original technician
  3. Customer Notifications: Only for final outcomes: Repair Fixed or Unfixable

P0

TICKET: DASH-005

Repair Dashboard - Scoped Visibility & Technician Transfer
Dashboard Permissions
Description

The original idea of a supervisor seeing all technicians' boards is scrapped. It creates visual clutter and exposes unnecessary information. A simpler, role-based view is adopted.

Requirements
  1. Personal Dashboard: Technician's "My Repairs" only shows repairs assigned to them
  2. Admin Global View: Admins have "All Repairs" list/table view (not Kanban)
  3. Transfer Function: Admin or current technician can transfer repair from Repair Detail View
